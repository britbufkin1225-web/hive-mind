# Phase 28D — Visual Direction Lock

Parent label: **devdevbuilds** · Phase type: **documentation / portfolio lock
only** (no frontend, CSS, backend, API, schema, package, runtime, or screenshot
changes).

This note locks the Hive|Mind visual/product direction after the Phase 28B
implementation and the Phase 28C evidence pass, so the portfolio-facing docs
present one consistent story before Phase 29 begins.

## Locked visual identity

Hive|Mind is a **graph-primary AI memory / intelligence workspace**:

- The **Knowledge Graph is the full application surface / viewfinder** — it
  fills the viewport edge-to-edge at every breakpoint.
- The shell is a **dark black/chrome/metal serious dev-tool shell** with
  minimal non-graph color; shell chrome carries no decorative accent color.
- There is **no persistent sidebar** as the default pattern.
- Graph visual energy is concentrated in **nodes, edges, groups, idle aura,
  selection pulse/glow, and future interaction polish** — not in the chrome.

Explicitly rejected patterns: **not dashboard-first, not card-grid-first, not
SaaS sidebar soup**.

## Accepted overlay model

Supporting tools are **contextual overlays summoned over the graph**, never
permanent dashboard columns:

- Vault / status summary
- Source Registry
- Intelligence Report
- Console
- Legend / groups / node-edge lists
- Selected-node inspector

Overlays are translucent, glass-like, non-occluding, and appear only when
summoned or selection-triggered. All surfaces stay read-only; nothing mutates
the graph, sources, or store.

## Portfolio-facing product language

The README leads with: deterministic **backend-derived intelligence surfaces**
(Dreaming Suggestions, Temporal Knowledge Decay, Provenance Chains, Query
Trails) over a Source Registry and Knowledge Graph — reproducible rule-based
derivation, honest empty states, **no AI/LLM calls**. Hive|Mind is a
human-directed devdevbuilds project that uses assisted development tooling
ethically — human review, validation, and merge control on every change — and
it does not claim production readiness or autonomous/unreviewed AI decisions.

## Guardrails for future frontend polish

- Planning (29A) lands and is reviewed before any implementation (29B).
- Polish stays inside the Phase 28A true graph-primary contract: full-viewport
  graph, contextual overlays, colorless chrome, no persistent sidebar.
- No backend/API/schema/package/dependency changes inside frontend polish
  phases; the graph stays read-only with no mutation paths.
- No fake liveliness, fake telemetry, or overlays not backed by real
  backend-derived data; honest empty states are preserved.
- Screenshot evidence is refreshed only in the dedicated QA pass (29C), with
  prior capture history preserved.
