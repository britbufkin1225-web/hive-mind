## 2. End-to-End Flow Summary

```
Backend (intelligence.py / provenance.py)
    ↓  derives ProvenanceChain records from HiveStore + SourceRegistry
GET /api/intelligence/report
    ↓  returns IntelligenceReport JSON including provenance_chains[]
Frontend (IntelligenceReportPanel.tsx)
    ↓  renders each ProvenanceChain as a read-only ProvenanceRow
User sees backend-derived, labeled, read-only provenance evidence
```

No fixture data is used for Provenance Chains in the demo-ready state.
No mutation controls exist at any layer.

## 3. What Is Implemented

### 3.1 Backend Derivation

Provenance Chains are generated by backend logic on every call to
`GET /api/intelligence/report`. The derivation path:

- `apps/backend/app/services/intelligence.py` — report assembly entry point.
- `apps/backend/app/services/provenance.py` — deterministic chain derivation
  from existing `HiveStore` nodes/edges and `SourceRegistry` source records.

Each `ProvenanceChain` record is derived from existing, already-captured fields:

| Field | Source |
|---|---|
| `node_id` | `HiveStore` node |
| `source_id` | Node's `source_id` reference |
| `source_type` | `SourceRecord.source_type` from registry |
| `origin_path` | Import `file_meta` / vault-relative path |
| `links` | Node's incident edge records |
| `linked_node_ids` | Neighbor node IDs from edges |
| `derived_edge_ids` | Edges where `metadata.origin == "knowledge_graph_builder"` |
| `stored_edge_ids` | Edges where origin is not derived |
| `last_imported_at` | `SourceRecord.last_imported_at` |
| `metadata.evidence` | Backend-owned evidence trail |
| `metadata.derived` | `true` — marks as backend-derived |

The derivation is deterministic: identical store/registry state produces
identical output. No AI/LLM inference is involved.

**Why backend-derived instead of frontend-only fixture?**
Frontend fixtures are static placeholders. Backend derivation makes the demo
data traceable to real application state owned by the backend. It makes the
provenance claim honest: when a chain record appears, it reflects an actual
source/import/node relationship, not copied sample text.

### 3.2 API Delivery

Provenance Chains are delivered through the existing
`GET /api/intelligence/report` endpoint. No new endpoint was added.

**Why no new endpoint?**
The Intelligence Report is already the consolidated delivery surface for all
read-only intelligence sections. Adding a standalone provenance endpoint would
create endpoint sprawl while the intelligence layer is still MVP/read-only. One
endpoint, one fetch, one error boundary in the frontend.

### 3.3 Frontend Visibility

The `IntelligenceReportPanel.tsx` component renders Provenance Chains via the
`ProvenanceRow` sub-component:

| UI element | Source field |
|---|---|
| Node ID (code tag) | `chain.node_id` |
| Source type tag | `chain.source_type` |
| Link summary | `{links.length} link(s) · {linked_node_ids.length} linked node(s)` |
| Origin path | `chain.origin_path` |
| Last imported | `formatDate(chain.last_imported_at)` |
| Section count badge | `report.summary.provenance_chain_count` |
| Derived badge | `isDerivedRecord()` on `metadata.derived === true` |
| Empty state | Clean "no data" message when graph has no traceable provenance |

**Why read-only display only?**
Provenance is an audit/traceability surface. It presents evidence of where
knowledge came from; it is not an editing surface. Adding mutation controls
here would conflate origin evidence with content management, which are
intentionally separate concerns in Hive|Mind's architecture.

**Why inside the Intelligence Report instead of a standalone dashboard section?**
Provenance Chains are one of several advisory intelligence surfaces. Keeping them
inside the Intelligence Report panel preserves the single-panel mental model and
avoids premature dashboard expansion before the intelligence layer is fully
stabilized.

## 5. Rationale Notes

### Why Provenance Chains are backend-derived

Frontend fixtures are useful for initial demo/screenshot readiness but cannot
make traceability claims credible. Backend derivation ensures that each chain
record corresponds to a real source/node/edge relationship in the store. It also
means the demo improves automatically as real data is imported, without any
frontend fixture maintenance.

### Why the feature remains read-only

Provenance is an audit surface. Its contract is to show evidence honestly. If
users could edit provenance records, the audit chain would become unreliable.
Read-only-first is the standing Hive|Mind intelligence-layer guardrail and is
especially important here because the data is already downstream of real import
events.

### Why Provenance Chains stay inside the Intelligence Report

The Intelligence Report is the deliberate consolidation point for all derived
intelligence sections. Separating Provenance Chains into a standalone dashboard
feature at this stage would expand the UI surface area without delivering
proportionate user value — the sections are most useful together as a unified
"what does the system know and how confident is it?" view.

### Why Query Trails are deferred

Query Trails require query-history persistence, which is a non-trivial storage
design decision. Implementing Query Trails without persistence would produce
demo fixtures only, which would be misleading as a "provenance" feature. The
deferral keeps the Provenance Chains section honest: what is shown is what the
backend can actually derive from existing records.

### Why no AI/LLM attribution is claimed

All chain records are derived from explicit `source_id`, `origin_path`,
`last_imported_at`, and edge-origin fields that already exist in the store.
Claiming AI attribution for this deterministic lookup would be inaccurate.
Hive|Mind's intelligence guardrail is "deterministic before clever" — AI/LLM
integration is a separately planned phase.

### Why QA evidence is documented before the next phase

Each intelligence surface has had its own QA/evidence pass (Phase 13C for
Temporal Decay, Phase 14E for Dreaming Suggestions, Phase 15E here). This
practice ensures that the implementation arc is closed with a clear record of
what was built, what was deferred, and why — before scope creep from the next
phase can obscure the boundary.
