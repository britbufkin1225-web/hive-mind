# Hive|Mind Demo Guide

This guide is for portfolio reviewers, demo prep, and screenshots. It explains
what Hive|Mind is, what is implemented now, and how to present the intelligence
surfaces honestly. For a section-by-section spoken walkthrough, see the
[Demo Script](demo-script.md); for what to capture, see the
[Screenshot Checklist](screenshot-checklist.md).

## Demo framing

Hive|Mind is a backend-focused knowledge graph project with a cybersecurity
learning angle. The core value is the pipeline:

1. Register or import knowledge sources.
2. Normalize source data into stable backend models.
3. Query and inspect the data through safe API/frontend surfaces.
4. Present the graph and future intelligence concepts without mutating data
   automatically.

The project is not an AI chatbot wrapper. It is a governed knowledge-system
foundation with clear contracts, read-only review surfaces, and room for future
deterministic intelligence.

## What is implemented now

- FastAPI backend with health/status routes.
- Local JSON-backed `HiveStore` persistence.
- Source Registry API and frontend panel.
- One-shot Obsidian markdown import.
- Safe Hive Console API and frontend panel.
- Knowledge Graph API and read-only frontend graph visualization.
- Intelligence Report API and frontend panel. The Temporal Decay section is
  backend-derived from real store timestamps (Phase 13A); Dreaming Suggestions
  are backend-derived from graph state (Phase 14C); Provenance Chains are
  backend-derived from existing source/import/node/edge records (Phase 15C);
  Query Trails remain deterministic demo fixtures.
- Agent Lab documentation for coordinating AI-assisted work without branch
  collisions.

## What is real vs. demo-only

In the Intelligence Report, **Temporal Knowledge Decay is real**: its rows are
derived read-only on the backend from each node's actual store timestamps, using
fixed thresholds (fresh <= 30 days, aging <= 90 days, then stale) and no AI.
Every row carries a plain-language "reason" explaining its classification.

**Dreaming Suggestions are real**: they are derived read-only from graph
nodes/edges using deterministic duplicate/orphan/stale-link rules. **Provenance
Chains are real**: they are derived read-only from existing source registry,
store source, node, import metadata, and edge records. Both carry explicit
backend evidence metadata and do not mutate the graph.

The Query Trails section is still **fixture-backed** — realistic but static
illustrative data, not generated by query persistence.

These fixtures are useful for showing the planned product direction. Do not
narrate them as live derived output.

## What is planned later

- Query trail persistence and review.
- Richer temporal-decay signals (e.g. last-referenced / last-seen) beyond the
  import/update timestamps used in the Phase 13A MVP.
- Read-only Agent Ops surfaces based on `docs/agent-lab/`.
- AI/LLM integration only after a separate plan and guardrail review.

## Run locally

Prerequisites: Node.js 20+ and Python 3.11+.

```bash
npm install
python -m venv .venv
python -m pip install -r apps/backend/requirements-dev.txt
```

Run the backend and frontend in separate terminals from the repository root:

```bash
npm run dev:backend
npm run dev:frontend
```

Open:

- Frontend: <http://localhost:5173>
- Backend health: <http://localhost:8787/api/health>
- Interactive API docs: <http://localhost:8787/docs>

> **Backend port:** `8787` is the canonical local dev backend port, and the
> frontend client (and `.env.example`) default to it. Always start the backend
> with `npm run dev:backend`. Launching `uvicorn` off-script binds its default
> port `8000` instead, so the frontend's `http://localhost:8787/api` requests
> hit nothing and surface as `Failed to fetch` — the cause of the Phase 20D
> connection failures.

## Validate before demo

```bash
npm run check:frontend
npm run check:backend
```

Manual checks:

- Backend connection says connected.
- Source Registry renders records and inspector details.
- Knowledge Graph renders nodes/edges and selected-item details.
- Intelligence Report renders the Temporal Decay section with its
  **Backend-derived** badge and per-row reason/age, Dreaming and Provenance with
  derived evidence, and Query Trails with **Demo data** badges; the whole panel
  stays read-only.
- Console accepts safe commands such as `help`, `status`, and `list nodes`.
- Unsafe shell-like commands are blocked by the console.

## How to explain the intelligence surfaces

Use this wording:

> The Intelligence Report is read-only. Temporal Knowledge Decay is real and
> backend-derived from store timestamps using fixed thresholds — no AI. Dreaming
> Suggestions and Provenance Chains are also backend-derived from existing graph,
> source, import, and edge records. Query Trails are still deterministic fixtures
> that show where Hive|Mind is headed.

Avoid saying:

- "The app uses AI to find connections."
- "Temporal decay uses AI or a scoring model." (It's plain timestamp thresholds.)
- "The provenance engine traces every fact."
- "Query trails are persisted."

## Demo order

1. Start with the README framing: source material becomes structured,
   inspectable graph data.
2. Show backend health and API docs to establish the FastAPI surface.
3. Show Source Registry to explain source governance.
4. Show Knowledge Graph to demonstrate normalized read-only graph inspection.
5. Show Intelligence Report: Temporal Decay as real backend-derived read-only
   logic, Dreaming/Provenance as derived evidence surfaces, and Query Trails as
   labeled future-facing fixtures.
6. Show the Console as a safe app command interface, not a shell.
7. Mention Agent Lab docs as the coordination layer for multi-agent development.
