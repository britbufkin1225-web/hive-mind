# Hive|Mind Demo Guide

This guide is for portfolio reviewers, demo prep, and screenshots. It explains
what Hive|Mind is, what is implemented now, and how to present the intelligence
surfaces honestly. For a section-by-section spoken walkthrough, see the
[Demo Script](demo-script.md); for what to capture, see the
[Screenshot Checklist](screenshot-checklist.md).

## Demo framing

Hive|Mind is a backend-focused knowledge graph project with a cybersecurity
learning angle. The core value is the pipeline:

1. Register or import knowledge sources.
2. Normalize source data into stable backend models.
3. Query and inspect the data through safe API/frontend surfaces.
4. Present the graph and future intelligence concepts without mutating data
   automatically.

The project is not an AI chatbot wrapper. It is a governed knowledge-system
foundation with clear contracts, read-only review surfaces, and room for future
deterministic intelligence.

## What is implemented now

- FastAPI backend with health/status routes.
- Local JSON-backed `HiveStore` persistence.
- Source Registry API and frontend panel.
- One-shot Obsidian markdown import.
- Safe Hive Console API and frontend panel.
- Knowledge Graph API and read-only frontend graph visualization.
- Intelligence Report API and frontend panel backed by deterministic demo
  fixtures.
- Agent Lab documentation for coordinating AI-assisted work without branch
  collisions.

## What is demo-only

The Intelligence Report is fixture-backed. It currently shows realistic sample
entries for:

- Dreaming-style suggestions.
- Temporal decay-style statuses.
- Provenance-style chains.
- Query trail-style history.

These entries are static illustrative data. They are useful for showing the
planned product direction, but they are not generated by real intelligence
logic.

## What is planned later

- Real deterministic Dreaming suggestions from graph/store state.
- Real temporal decay calculations from timestamps and source context.
- Real provenance chains from source/import/node relationships.
- Query trail persistence and review.
- Read-only Agent Ops surfaces based on `docs/agent-lab/`.
- AI/LLM integration only after a separate plan and guardrail review.

## Run locally

Prerequisites: Node.js 20+ and Python 3.11+.

```bash
npm install
python -m venv .venv
python -m pip install -r apps/backend/requirements-dev.txt
```

Run the backend and frontend in separate terminals from the repository root:

```bash
npm run dev:backend
npm run dev:frontend
```

Open:

- Frontend: <http://localhost:5173>
- Backend health: <http://localhost:8787/api/health>
- Interactive API docs: <http://localhost:8787/docs>

## Validate before demo

```bash
npm run check:frontend
npm run check:backend
```

Manual checks:

- Backend connection says connected.
- Source Registry renders records and inspector details.
- Knowledge Graph renders nodes/edges and selected-item details.
- Intelligence Report renders every fixture section and remains clearly
  read-only/demo-oriented.
- Console accepts safe commands such as `help`, `status`, and `list nodes`.
- Unsafe shell-like commands are blocked by the console.

## How to explain the intelligence surfaces

Use this wording:

> The Intelligence Report is currently a deterministic fixture demo that shows
> where Hive|Mind is headed. The contracts and UI exist, but real Dreaming,
> temporal decay, provenance inference, query memory, and AI/LLM logic are future
> work.

Avoid saying:

- "The app uses AI to find connections."
- "Temporal decay is implemented."
- "The provenance engine traces every fact."
- "Query trails are persisted."

## Demo order

1. Start with the README framing: source material becomes structured,
   inspectable graph data.
2. Show backend health and API docs to establish the FastAPI surface.
3. Show Source Registry to explain source governance.
4. Show Knowledge Graph to demonstrate normalized read-only graph inspection.
5. Show Intelligence Report as a future-facing fixture demo.
6. Show the Console as a safe app command interface, not a shell.
7. Mention Agent Lab docs as the coordination layer for multi-agent development.
